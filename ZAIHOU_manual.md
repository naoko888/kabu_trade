## 1. 起動時

### ① データ準備
- CPI イベントカレンダーを読み込む（シグナル判定で使用）
- `open_positions.json` から前回保有していたポジション情報を読み込む
- `micro_dry_log_all.csv` から当月の損益を集計し、月次 DD 停止状態を復元する
- `micro_5min.csv` から過去 300 本分の 5 分足データを読み込む（ウォームアップ）
  - データが 300 本未満の場合はエントリーを無効化して起動する

### ② API 接続
- kabuステーション API にパスワードでトークンを取得する
- 銘柄コードを取得する（NK225マイクロ先物のみ。取得失敗時は銘柄登録も失敗し起動を中止する）
- 翌限月の銘柄コードも取得し、バー収集対象として登録する
- 銘柄を API 登録銘柄リストに登録する（失敗時は起動を中止する）
- WebSocket を起動し、リアルタイム価格配信を受信開始する

### ③ ポジション復元（本番モードのみ・ブローカーAPIを正とする）
- ブローカーの `/positions` API を取得し、実際の建玉一覧を正とする
- コンテキストファイル（`open_positions.json`）を補助情報として照合：
  - HoldID が一致 → 復元（エントリー価格はブローカー値を使用）
  - コンテキストにあってブローカーにない → 「停止中に約定済み」として破棄・Discord 通知
  - ブローカーにあってコンテキストにない → 「不明ポジション」として監視不可・Discord 警告
  - HoldID なし（DRY_RUN で作られたポジション）→ 復元しない（Bug4 解決）
- DRY_RUN モードでは `open_positions.json` からそのまま読み込む（従来通り）

### ④ TP 再発注（本番モード・ポジションあり時のみ）
- 保有中のポジション全件に対し、現在のセッションに合わせた **TP 指値注文** を再発注する
- SL はソフトウェア監視のため再発注不要

## 2. メインループ（0.4 秒ごと）

起動完了後は以下を繰り返す。

- 板情報（現在値・Bid・Ask）を取得する
- ポジションがあれば → ポジション監視を実行する（→ 3章）
- エントリー条件を判定する（→ 4章）
- 5 分ごとに 5 分足 CSV を保存する
- 16:45・8:00 にセッション切替の TP 再発注を行う（SL はソフトウェア監視のため不要）
- 毎時 0 分と 8:30・16:45 に Discord へポジション状況を報告する
- 毎時 0 分にハートビート通知を Discord へ送る

---

## 3. ポジション保有中の監視

ポジション保有中は 0.4 秒ごと（WebSocket tick 受信時も）に以下を確認する。

板情報から現在値を取得し、各ポジションについて順番に判定する。

### TP 到達
- long ポジションで `現在値 >= TP価格 + 5円`、または short で `現在値 <= TP価格 - 5円`
- 本番モード: `/positions` でポジションが消滅していれば TP 約定済みと判断し損益を記録する
- ポジションがまだ残っていれば次の tick で再確認する

### SL 到達（ソフトウェア監視）
- long ポジションで `現在値 <= SL価格`、または short で `現在値 >= SL価格`
- **SL はブローカー注文なし。ソフトウェアが価格を監視し、到達時に成行決済を発注する**
- 本番モード:
  1. **ブローカー事前確認**: `_is_position_already_closed(hid, tp_oid)` で `/positions` と `/orders` を確認する。ポジション消滅 または TP 約定済みが確認できた場合は TP 約定済みとして記録して終了（成行発注しない）
  2. TP キャンセル → `_wait_for_hold_release()` で HoldQty==0 を確認
     - `None` 返り（ポジション消滅）→ TP 約定済みとして記録して終了
     - `False` 返り（タイムアウト）→ Discord 最緊急アラート（手動決済要）
  3. HoldID が取得済みの場合: `ClosePositions` 指定の成行発注（FrontOrderType=120, TradeType=2）
  4. HoldID なしの場合: FIFO 成行発注
  5. 約定確認後 → 実約定価格で損益記録
  6. 約定未確認・発注失敗 → Discord 最緊急アラート（手動決済要）
- スリッページ: ポーリング間隔（最大 0.4 秒）分の遅延。N225マイクロ先物では概ね 1〜2 ティック（5〜10 円）程度

## 4. エントリー判定

毎ポーリングで確認バー（最新の確定 5 分足）が更新されていれば以下を実行する。

### スキップ条件（いずれかに該当する場合はエントリーしない）
- 月次 DD 停止中
- 同一バー時刻でシグナルチェック済み（重複防止）
- バーが前のセッションのもの（セッション跨ぎ）
- バーが現在時刻より 10 分以上古い（陳腐化）
- 金曜 5:00〜6:00 のエントリー禁止時間帯
- **系統③: 5時台エントリー除外**（DST・冬時間ともに除外。EV+1.35pt / PF1.071 と弱く件数 1,267 件あるが損益への貢献が小さいため。2026-06-10 適用。付録H参照）
- **物理金曜 21:25〜21:35: 系統① エントリーブロック**（NFP 等米国経済指標が JST 21:30 に発表されるため。BT 上の損益改善は微小だが実測スリッページが信頼できないため安全側に倒す。実装: zh_entry.py `fri_2130_entry_block`）
- ウォームアップ残本数が残っている（最大 26 本）

### シグナル判定
- 系統①③: `check_s1_s3()` でシグナルを確認する（①=long、③=short）
- 系統④: `ENABLE_S4=True` の場合のみ `check_s4()` を実行する（long のみ）
- 系統⑤: `ENABLE_S5=True` の場合のみ `check_s5()` を実行する（short のみ）

> **系統④⑤の ON/OFF**: `zh_config.py` の `ENABLE_S4` / `ENABLE_S5` を `True` / `False` で切り替える。  
> 再起動不要（次の check_entry サイクルから反映）。  
> **現在: 両方 `False`（停止中）**

### エントリー実行（シグナルあり）
1. 同じ系統に逆方向ポジションがあれば成行で決済する（逆ポジ決済）
2. エントリーを実行する（→ 5章）

## 5. エントリー実行

### DRY_RUN モード
実際の発注は行わず、現在の板価格でポジションをリストに追加して終了する。

### 本番モード
1. エントリー前に `/positions` を確認して既存建玉の ID 一覧を取得する  
   取得失敗時はエントリーをスキップする（機会損失を許容）
2. 成行発注する（FrontOrderType=120, TimeInForce=FAK）
3. 約定確認をポーリングする（最大 10 秒）  
   タイムアウト時は注文をキャンセルしてスキップする
4. 実約定価格をもとに SL 価格・TP 価格を再計算する
5. `/positions` を再取得して新規建玉の HoldID（ExecutionID）を特定する
6. **SL はソフトウェア監視のみ（ブローカー注文なし）**
7. TP 指値注文を発注する（FrontOrderType=20, TimeInForce=FAS, HoldID 指定）
8. TP 注文の受付状態を確認する（非アクティブ確定時は Discord 警告）
9. ポジションをリストに追加して `open_positions.json` に保存する

## 6. TIME 決済（MAX_HOLD）

エントリーから経過した 5 分足の本数が系統ごとの上限を超えた場合に決済する。

- 本番モード: SL・TP 注文を両方キャンセルしてから成行返済する
  - キャンセルが一方でも失敗した場合は緊急フラット（→ 8章）を実行する
  - キャンセル後にポジションが消滅していれば「SL/TP が先に約定済み」と判断しスキップする
  - 成行返済の約定が確認できなかった場合は Discord に緊急警告を出す

---

## 7. 強制決済

以下の条件のいずれかを満たす場合に強制決済を行う（TIME 決済と同じ手順）。

| 条件 | 対象 |
|---|---|
| 現在時刻が **15:30**〜17:00 または **5:40**〜8:45 | 全系統（セッション終了） |
| 土曜 6:00 以降 | 全系統（金曜夜間終了） |
| 23:50 以降 | 系統①のみ（夜間終了） |

セッション終了強制決済は、トリガーされた時刻を記録して 1 回のみ実行する。

> ⚠️ **暫定措置中（Bug8 検証期間）**: 通常は 15:40 / 5:55 だが、2026-06-05 より 15:30 / 5:40 に前倒し中。解除方法は「15章 暫定措置」を参照。

---

## 8. 緊急フラット

SL・TP のキャンセルに失敗した場合に実行する最終手段。

1. `/positions` でブローカー側のネットポジション（買い枚数－売り枚数）を確認する
2. ネットポジションが 0 でなければ逆方向に成行発注してフラット化する
3. 発注失敗時は「手動決済要」として Discord に最緊急アラートを送る

---

## 9. TP 再発注（セッション切替）

先物は日中・夜間でセッション（Exchange コード）が異なり、注文はセッション内でのみ有効。  
**SL はソフトウェア監視のため再発注不要。TP 注文のみ再発注する。**

- **16:45**（夜間セッション前気配）: 全ポジションの TP を Exchange=24 で再発注
- **8:00**（日中セッション前気配）: 全ポジションの TP を Exchange=23 で再発注

手順: 旧 TP 注文キャンセル → 新 TP 注文発注 → 受付確認（失敗時は Discord 警告）

---

## 10. DD 停止

月次損益（円）が DD 上限（デフォルト -300,000 円）を下回った場合。

- `monthly_stopped = True` に設定し、以降のエントリーを全系統でブロックする
- ポジション決済は引き続き実行する（保有中ポジションは監視・決済を継続する）
- 翌月 1 日以降の最初のバー確認時に月次リセットして自動再開する

---

## 11. ブローカー整合チェック（reconcile）

本番モードでポジションがある場合、60 秒ごとに内部状態とブローカーを突合する。

| 不一致パターン | 対応 |
|---|---|
| ブローカー側にポジションなし | SL/TP キャンセル → 内部ポジションをクリア |
| 内部と逆方向のポジション | 緊急成行フラット |
| 枚数が合わない | Discord 警告のみ |

---

## 12. 異常時の動作

### 認証エラー（401）
- 自動でトークンを再取得し、銘柄を再登録して直前のリクエストをリトライする
- 再取得のクールダウン: 10 秒
- 認証エラーが **5 回連続** した場合は安全終了する

### 通信エラー（タイムアウト・接続失敗）
- リクエストは None を返す
- 各処理がそれぞれの判断で継続・スキップ・再試行を選択する

### HoldID 取得失敗
- SL/TP 注文を `ClosePositionOrder=0`（FIFO）で代替発注する
- 複数ポジション保有時に意図しない建玉が返済されるリスクがある

---

## 13. 終了時

### 正常終了（週末）
- 土曜 6:00 以降にポジションが残っていれば強制決済を実行する
- 5 分足データを CSV に保存する
- Discord に「週末終了」と本日の損益を通知する

### 手動停止（Ctrl+C）
- 5 分足データを CSV に保存する
- Discord に「手動停止」を通知する
- ポジションは `open_positions.json` に保存済みのまま残る（次回起動時に復元）

### 例外終了
- スタックトレースをログと Discord に送信する
- ポジションは保存済みのまま残る

## 15. 暫定措置（Bug8 検証期間中）

> **適用期間**: 2026-06-05〜  
> **注**: Bug7・Bug8 は 2026-06-05（commit:824d393）で解決済み。暫定措置は実機検証が完了したら解除可。

### 背景

強制決済パス（セッション終了・時間切れ等）に、SL と同じ「決済前にTP予約解放を確認する処理」が未実装（Bug8）。  
FIFO（名指しなし）決済で予約が残ったまま失敗すると無限リトライになる可能性があった。  
コード修正（commit:824d393）で根本対処済みだが、実機で動作確認が取れるまで暫定措置を継続する。

### 変更内容

| 項目 | 通常 | 暫定措置中 |
|---|---|---|
| 日中強制決済 | 15:40 | **15:30** |
| 夜間強制決済 | 5:55 | **5:40** |
| 日中エントリー禁止 | なし | **15:30 以降** |
| 夜間エントリー禁止 | 金曜 5:00〜6:00 のみ | **毎日 5:40 以降** |

### 解除方法

以下の 2 ファイルを元に戻して git commit する。

**zh_monitor.py**（2 箇所）:
- `1530 → 1540`、`540 → 555`

**zh_entry.py**（2 行を削除）:
- `if 1530 <= hhmm_now < 1700:` の行
- `if 540 <= hhmm_now < 845:` の行

- **global 宣言に注意**: Python の `global` は dotted name（`zh_bar.foo` 等）を受け付けない。replace_all で変数名を置換する際は事前に `global` 宣言行を確認して手動修正する。
- **zh_monitor.py が ZAIHOU_signals に依存する理由**: `_SL_MAP` / `_TP_MAP` / `_MH_MAP` の参照のため。Phase 7 以降で zh_entry.py が持つべき情報だが、現時点では zh_monitor.py が保有（設計懸念 D1）。
- **`_wait_for_hold_release(hid, max_retries=10, interval=0.3)` の役割**: SL 到達時に TP キャンセル後の HoldID 解放を確認するヘルパー関数（zh_monitor.py）。`/positions` の `HoldQty==0` を最大 10 回ポーリング（最大 3 秒）。戻り値の意味（Bug9 修正後）: `True`=HoldQty==0 解放済み→ClosePositions 送信可 / `None`=ポジション消滅（TP 約定済み）→ClosePositions 不要 / `False`=タイムアウト→手動対応要。呼び出し元は `is False` で厳密に判定すること（`not False` では None との区別ができない）。J3-A 実装時はこの関数ごと移植する。
- **`_is_position_already_closed(hid, order_id)` の役割**（Bug9 対応。2026-06-10 追加）: SL 発動前にブローカー状態を確認し、ポジションが既にクローズ済みかを返すヘルパー関数（zh_monitor.py）。① `/positions` で `LeavesQty>0` のポジション存在確認 → ② `/orders` で `State==5 かつ CumQty>0`（全約定）確認。**設計原則: アクション前にブローカーの現在状態を確認する。** 現在は SL 発動前（`order_id=tp_oid`）で使用。将来 SL がブローカー逆指値注文になった場合は TP 発動前（`order_id=sl_oid`）でも共通使用できる。
- **`_adjust_trading_day` による曜日判定の動作**: バーの datetime は `_adjust_trading_day` で補正済み（17:00以降 → 翌取引日付）。シグナル判定の曜日・時刻フィルターはこの補正後 datetime を使用するため、**夜間バーの weekday は物理日ではなく翌取引日の weekday** になる。例: 2026-06-02（火）17:00 のバーは 2026-06-03（水）17:00 として扱われ、③ の S3_WEEKDAYS 判定が水曜（wd=2）で行われる。これは設計通りの動作（trading day convention）。BT 側のパフォーマンス比較スクリプトがこの変換を考慮していなかったため、2026-06-03 に `micro_performance_summary.py` の照合キーを trade_date ベースに修正した。

## 背景：先物の「取引日」とは

N225マイクロ先物の取引日は **カレンダー日付とズレる** ことがある。

```
例：5/27（月）のナイトセッション
  物理時刻: 2026-05-27 17:00 〜 2026-05-27 23:55
  取引日  : 2026-05-27（月）の取引として扱われる

  ただしナイトセッションの続き（深夜〜翌朝）
  物理時刻: 2026-05-28 00:00 〜 2026-05-28 05:55
  取引日  : 2026-05-27（月）の取引として扱われる（翌日ではない）
```

**正しい取引日順のバー並び順：**
```
5/27 17:00 → 5/27 23:55 → 5/28 00:00 → 5/28 05:55 → 5/28 08:45 → 5/28 15:15
  ↑ナイト開始          ↑深夜              ↑翌朝日中
  ←─────── 5/27の取引日 ──────────────────────────→
```

---

```
修正前（物理時間順）:
  5/27 00:00 → 5/27 08:45 → 5/27 15:15 → 5/27 17:00 → 5/27 23:55 → ...
  ← 日中 ─────────────────────→ ← ナイト →

修正後（取引日順）:
  5/27 17:00 → 5/27 23:55 → 5/28 00:00 → 5/28 08:45 → 5/28 15:15 → 5/28 17:00
  ← ナイト ─────────────────────────────────── ← 翌日中 →
```

これにより、系統④⑤の **移動平均・MACD などの指標値がライブシステムと乖離** していた。

## 修正箇所（`backtest_system45_combined.py`）

# 修正前
def _trading_day_sort_key(dt):
    if dt.hour < 17:
        trading_date = (dt - pd.Timedelta(days=1)).date()

## 付録D: BTデータ品質・精度調査（2026-06-08）

---

### D1. backtest_combined_all.py と micro_performance_summary.py の数値差

#### 症状

同じXLSXを使っているはずなのに、同月の損益が一致しないことがある。

```
例（2026年6月）:
  backtest_combined_all.py        → +13,788円
  micro_performance_summary.py    → +40,154円
```

#### 原因

`backtest_combined_all.py` は結果を `bt_result_combined.txt` に出力・保存する。  
**このファイルは実行した時点のXLSXデータを元にしたスナップショット**であり、その後XLSXが更新されても自動で更新されない。

```
bt_result_combined.txt 最終更新: 2026-06-04（4日前に生成）
N225microf_2026.xlsx  最終更新: 2026-06-08（今日更新）
→ 4日分のバー・シグナルの差が生じていた
```

`micro_performance_summary.py` は実行のたびに最新のXLSXを読み直すため、常に最新の数値を返す。

#### 対処

数値を比較・参照する際は `backtest_combined_all.py` を **その場で再実行して** `bt_result_combined.txt` を最新化すること。古いファイルを参照しない。

---

### D2. BT XLSX と BT CSV の数値差（micro_performance_summary.py内）

#### 症状

`micro_performance_summary.py` 内でXLSXとCSVを同じBTエンジンで実行しても損益が一致しない。

```
例（2026年6月）:
  BT XLSX → +40,154円
  BT CSV  → +52,526円
```

#### 原因

`micro_5min.csv` の一部バーで **O=H=L=C**（4本値がすべてClose価格と同一）になっているバーが多数存在する。

```
例（2026-06-04 02:00:00）:
  XLSX: O=68,235  H=68,235  L=68,150  C=68,160  （85pt レンジ）
  CSV:  O=68,160  H=68,160  L=68,160  C=68,160  （レンジ 0pt）
```

6/4以降で265本のバーに差異が発生（最大305pt差）。  
CSVはバー終了時点のClose 1点しか記録できておらず、期間中の High / Low が実際の値動きを反映していない。

#### BT精度への影響

BTのTP/SL判定はHigh/Lowを使う。

- **XLSX（正確）**: 正しいH/Lで本来のTP/SL到達を再現できる
- **CSV（不正確）**: H=L=Cのため急変動が見えず、SLが当たらなく見える → 見かけ上の損益が高くなる

**BTの基準データはXLSXを使うこと。CSVのBT結果は信頼性が低い。**

#### 根本原因（要調査）

`zh_bar.py` のバー構築ロジックが、5分間の実際のOHLを集計せず、バー終了時点の価格のみを記録している可能性がある。6/4以降に突然発生していることから、kabustation側のデータ受信形式の変化か、バー更新ロジックのバグが疑われる。今後の課題として調査が必要。

### D3. TP/SL 同バー問題の調査（analyze_tp_sl_ambiguity.py）

#### 問題の概要

BTは5分足のバーを1本ずつ走査してTP/SLを判定するが、**同じバーでHighがTP水準に達し、かつLowがSL水準に達した場合**、どちらが先に来たかはティックデータなしには判別できない。

現行BTはTP優先（楽観的仮定）を採用している。

```python
# 現行BTの判定順（ロングの例）
if High >= EP + TP:   # ← TP を先にチェック → TP 扱い
    return "TP"
if Low  <= EP - SL:   # ← SL は後
    return "SL"
```

#### 調査方法（`analyze_tp_sl_ambiguity.py`）

1. BT実行 → result="TP" のトレードを抽出
2. 各TP成立トレードについて、エントリーバーから1本ずつ走査してTPバーを特定
3. そのTPバーで SL条件（Low <= EP - SL など）も成立しているか確認
4. 成立していれば「SLが先だった可能性あり（曖昧）」としてカウント・損益差を集計

#### 調査結果（2026-06-08 XLSX全期間）

| 系統 | TP件数 | 曖昧件数 | 曖昧率 | 最大影響額 | TP+SL必要レンジ |
|---|---|---|---|---|---|
| ① | 683 | 1 | 0.15% | -3,000円 | 300pt以上 |
| ③ | 1,414 | 5 | 0.35% | -15,000円 | 300pt以上 |
| ④ | 93 | 2 | 2.15% | -9,600円 | 480pt以上 |
| ⑤ | 297 | 0 | 0.00% | 0円 | 380pt以上 |
| **合計** | **2,487** | **8** | **0.32%** | **-27,600円** | |

- 曖昧バーの最小レンジ: 385pt、最大レンジ: 695pt
- 「最大影響額 -27,600円」は8件すべてをSL扱いにした場合の最悪ケース。実際はTP/SLどちらが先かは不明

#### 結論

**実務上問題なし。修正不要。**

- 3年間・全TPトレードで8件（0.32%）のみ
- 最大影響額 -27,600円は全期間損益の誤差の範囲
- バーレンジ300pt以上という稀なケースでのみ発生（急変動日・CPI発表等）

### E1. データ読み込みの正しい手順

#### Excel の日付フォーマット

`N225microf_xxxx.xlsx` の日付列は **取引日（翌営業日）ベース** で格納されている。

```
物理時刻: 2026-05-27 17:05
Excel格納: date=2026-05-28, time=17:05
→ datetime = date + time で構築すると「2026-05-28 17:05」になる
```

そのため `sort_values("datetime")` だけでは正しい物理順にならない。

```
誤ったソート結果:
  2026-05-27 08:45 → 2026-05-27 15:15 → 2026-05-28 08:45 → 2026-05-28 17:05
  ↑夜間セッション(2026-05-28 17:05)が翌日中の後ろに来てしまう
```

#### 正しいソートコード（必須）

```python
df["_tday"]     = df["datetime"].dt.date
df["_sort_grp"] = (df["datetime"].dt.hour < 17).astype(int)  # 夜間=0, 日中=1
df = (df.sort_values(["_tday", "_sort_grp", "datetime"])
        .drop(columns=["_tday", "_sort_grp"])
        .reset_index(drop=True))
```

> ⚠️ **テクニカル指標（MA・ATR・MACD等）の計算は必ずこのソート後に行うこと。**  
> ソート前に計算すると指標値がライブシステムと乖離する（付録B ソートバグ修正参照）。

---

### E2. 絶対条件（破ってはいけないルール）

#### 条件1: エントリーはシグナルバーの**翌バーopenで成行**

```python
# シグナル確定: df_confirmed = df.iloc[:-1] の最終バー
# エントリー価格: 次のバー（df.iloc[-1]）の open
ep = next_bar["open"]
```

シグナルバー自体の close でエントリーすると将来データを使うことになる（先読みバイアス）。

#### 条件2: 同一バー内 TP/SL 両触れ → **SL 優先**（悲観的）

```python
if side == "long":
    tp_hit = hi >= ep + tp
    sl_hit = lo <= ep - sl
    if tp_hit and sl_hit:   # 同バーで両方触れた場合
        exit_pnl = -sl; exit_tag = "SL"; break   # ← SL 優先
    if tp_hit:
        exit_pnl =  tp; exit_tag = "TP"; break
    if sl_hit:
        exit_pnl = -sl; exit_tag = "SL"; break
# short も同様（tp_hit = lo<=ep-tp, sl_hit = hi>=ep+sl）
```

> ⚠️ TP 優先（楽観的）で実装すると PF が水増しされる。  
> 付録D3 の調査では 0.32% の件数に影響し最大 -27,600円 の差が生じる。

#### 条件3: セッション境界で強制決済（GAP_BOUNDS）

```python
GAP_BOUNDS = frozenset({1540, 555})  # hhmm（日中終了: 15:40、夜間終了: 5:55）
# バー走査中にこの時刻を跨いだら TIME 決済扱い
if bar_hhmm in GAP_BOUNDS:
    exit_pnl = bar["close"] - ep if side == "long" else ep - bar["close"]
    exit_tag = "TIME"; break
```

日中セッション終了（15:40）・夜間セッション終了（5:55）を跨いでポジションを持ち越さない。  
実運用の強制決済タイミングと合わせること。

#### 条件4: TP/SL 判定は **High/Low** を使う

```python
# 正しい判定
tp_hit = bar["high"] >= ep + tp   # ロング TP
sl_hit = bar["low"]  <= ep - sl   # ロング SL

# 誤り（Close のみ使用すると SL が当たらなく見える）
# tp_hit = bar["close"] >= ep + tp  ← NG
```

付録D2 の調査で CSV データでは H=L=C のバーが多数あり、SL 見落としによる PF 水増しが発生した。  
**BT の正解データは XLSX を使うこと（CSV は信頼性が低い）。**

### G5. コピペ問題と回避策（重要）

**症状**: RDP経由でローカルPCからVPS内のPowerShellにテキストをペーストすると、余計な文字や画像データが混入する。

**試したが解決しなかった方法:**

- RDP切断・再接続

**有効な回避策: Googleスプレッドシートを経由する**

### G6. Pythonファイル編集の注意事項（文字コード）

**症状**: PowerShellの `Set-Content` でPythonファイルを上書きすると文字コードがUTF-16になり、Pythonが `SyntaxError: (unicode error) 'utf-8' codec can't decode byte` で起動できなくなる。

**原因**: PowerShell 5.1 の `Set-Content` はデフォルトでUTF-16 LEで保存する。

**絶対にやってはいけない操作:**
```powershell
# これをやると.pyファイルがUTF-16になってPythonが読めなくなる
(Get-Content xxx.py) -replace '...' | Set-Content xxx.py
```

**正しいPythonファイルの編集方法（メモ帳を使う）:**
1. メモ帳を開く
2. ファイル → 開く → 対象の `.py` ファイルを開く
3. 編集する
4. **ファイル → 名前を付けて保存**（上書き保存ではない）
5. ファイルの種類: `すべてのファイル (*.*)`
6. エンコード: **UTF-8**（UTF-8 BOM ではない）
7. 保存
